# Prompt

Prompt is a minimal anonymous writing app. Visitors can write a text post, choose whether it is public or link-only, and share it through a human-readable URL. The app also includes a small admin area for moderation, IP bans, spam limits, database backups, credential management, and Google Analytics configuration.

The project is designed to run as a self-contained Next.js application backed by a local SQLite database.

## Features

- Anonymous post creation with an optional title and text content.
- Public and link-only visibility modes.
- Shareable five-word slugs generated from a database counter.
- Public reading navigation for previous, random, and next public posts.
- Direct post pages with view and share tracking.
- Public report flow with duplicate protection and rate limiting.
- Admin dashboard for post search, filtering, pagination, and moderation.
- IP bans and configurable spam-prevention settings.
- Admin-managed Google Analytics settings.
- SQLite database download and restore controls.
- Dockerfile and Coolify startup script for production deployment.

## Stack

- Next.js 16 App Router
- React 19
- TypeScript 6
- Tailwind CSS 4
- Prisma 7 with `@prisma/adapter-better-sqlite3`
- SQLite
- Server Actions for form mutations
- ESLint with Next.js core web vitals and TypeScript rules
- Node 22 for the Docker image

The UI uses small local components in `components/` and `components/ui/`, with Tailwind theme tokens defined in `app/globals.css`.

## Project Structure

- `app/` contains App Router pages, layouts, API routes, and server actions.
- `components/` contains public UI, post controls, admin UI, and shared primitives.
- `lib/` contains data access, authentication, slug generation, spam settings, app settings, request IP handling, and database-file helpers.
- `prisma/` contains the schema, migrations, and seed script.
- `generated/prisma/` is generated by Prisma and ignored by ESLint.
- `scripts/` contains operational scripts such as Coolify startup, password reset, and smoke checks.

## Environment

Create `.env` from the example file:

```bash
cp .env.example .env
```

Supported variables:

- `DATABASE_URL`: SQLite file URL. Local default is `file:./dev.db`.
- `PROMPT_ADMIN_SETUP_SECRET`: Secret required when creating the first admin account. Set this to a long random value before exposing the app publicly.
- `PROMPT_TRUST_PROXY_HEADERS`: Set to `true` only when the app is behind a trusted proxy or CDN that strips client-supplied forwarding headers.
- `NEXT_PUBLIC_TURNSTILE_SITE_KEY`: Optional Cloudflare Turnstile site key shown on the post form.
- `TURNSTILE_SECRET_KEY`: Optional Cloudflare Turnstile secret key used to verify the captcha token on submission.
- `NODE_ENV`: Set by Next.js or the deployment environment. Production sessions and report cookies use secure cookies when this is `production`.
- `PORT`: Used by deployment startup. Defaults to `3000` in the Docker image and Coolify script.
- `BIND_HOST`: Used by `scripts/coolify-start.sh`. Defaults to `0.0.0.0`.

Do not enable `PROMPT_TRUST_PROXY_HEADERS` on an untrusted direct internet deployment. The app uses forwarded IP headers for posting limits, reports, and IP bans only when this variable is explicitly enabled.

- `npm run build`: build the production app and run TypeScript checks.
- `npm run start`: start the built Next.js app.
- `npm run lint`: run ESLint.
- `npm run prisma:generate`: generate the Prisma client.

## Public Routes

- `/`: writing screen with a rotating prompt, optional title, content field, visibility selector, and navigation controls.
- `/p/[slug]`: public post page. Hidden posts return not found. View count increments when a post page is opened.
- `/read`: redirects to the most recent public post, or back to `/` if there are no public posts.
- `/about`: project description page.

## API Routes

- `GET /api/posts/viewer`: returns a random post, a specific post by slug, or an adjacent post using `slug` and `direction` query parameters.
- `POST /api/posts/[slug]/share`: increments the share count for a post.
- `POST /api/posts/[slug]/report`: records a report, deduplicates repeat reporters, rate-limits reports, and can auto-hide posts when the configured threshold is reached.

These API routes are used by the UI and are not intended as a stable public API contract.

## Admin Setup

On a fresh install, visit `/admin/setup` and create the first admin account. In production, `PROMPT_ADMIN_SETUP_SECRET` must be configured and supplied during setup.

After setup, admins sign in at `/admin/login`. Admin sessions are stored in an HTTP-only `prompt_admin_session` cookie and backed by the `AdminSession` table. Session tokens are stored in the database as SHA-256 hashes. Passwords are hashed with Node's built-in `scrypt`.

Changing the admin email or password clears all sessions for that admin, including the current session, so sign in again after saving.

## Admin Management

The admin dashboard at `/admin` supports:

- Searching posts by ID, slug, title, or content.
- Filtering by IP address, visibility, and report status.
- Sorting by newest, oldest, reports, visibility, or IP.
- Viewing post IDs, slugs, created dates, IP addresses, visibility, report counts, and previews.
- Hiding, restoring, deleting, clearing reports, and banning IPs.

The IP bans page at `/admin/ip-bans` supports:

- Adding an IP ban with an optional reason.
- Removing an existing IP ban.
- Blocking future posts from banned IPs when request IP capture is available.

The spam settings page at `/admin/spam-settings` controls:

- Maximum posts per IP per day. Use `0` to disable the daily cap.
- Cooldown between posts in seconds. Use `0` to disable cooldown.
- Duplicate text window in hours. Use `0` to disable duplicate blocking.
- Auto-hide report threshold. Use `0` to disable automatic hiding.

Defaults are 10 posts per day, 60 seconds between posts, a 24-hour duplicate window, and auto-hide after 3 reports.

The settings page at `/admin/settings` supports:

- Changing the admin email.
- Changing the admin password.
- Enabling or disabling Google Analytics.
- Saving a GA4 measurement ID such as `G-XXXXXXXXXX`.
- Downloading the current SQLite database.
- Uploading a SQLite database to replace the current database.

Database uploads validate the SQLite file header before replacement. After a restore, the app redirects to admin sign-in because the restored database may not contain the active session.

## Google Analytics

Google Analytics is configured in the admin settings page, not through an environment variable. Enter a GA4 measurement ID and enable analytics. When enabled, the root layout injects the `gtag.js` script after hydration. When disabled or no measurement ID is saved, no Google Analytics script is rendered.

Measurement IDs are normalized to uppercase and validated against the GA4 `G-...` format.

## Moderation and Visibility

Posts can have one of three visibility states:

- `public`: readable by direct URL and eligible for public previous, random, and next navigation.
- `link-only`: readable by direct URL but excluded from public navigation.
- `hidden`: unavailable to public readers and removed from navigation.

When an admin hides a post, Prompt stores the prior public state. Restoring the post brings it back as either `public` or `link-only`, depending on its previous visibility.

Reports are tracked in `PostReport` and deduplicated per reporter. If trusted IP headers are enabled, reporters are keyed by hashed IP address. Otherwise, Prompt uses an HTTP-only reporter cookie. Reports are limited to 10 per reporter per hour.

## Posting and Anti-Spam Behavior

Post content is normalized by trimming whitespace and removing null bytes. Empty posts are rejected. Titles are optional and capped at 120 characters.

If `NEXT_PUBLIC_TURNSTILE_SITE_KEY` is set, the write form renders a Cloudflare Turnstile challenge and post creation requires a valid captcha token verified using `TURNSTILE_SECRET_KEY`.

The code defines `MAX_POST_LENGTH` as 2,000 characters, but the active content-length check is currently disabled so long-form posts can be tested.

When a post is created, Prompt applies the configured spam settings to the captured IP address. If trusted IP headers are disabled or unavailable, posts are grouped into an unknown-IP bucket so rate limits still apply instead of being skipped.

Generated slugs are deterministic five-word phrases. They are backed by the `SlugCounter` table rather than random strings. The current word set supports more than 34 billion unique slugs.

## Database

Prompt uses SQLite through Prisma. The schema includes:

- `Post`: post content, slug, visibility, IP address, counts, and report metadata.
- `SlugCounter`: monotonic counter for slug generation.
- `AdminUser`: admin credentials.
- `AdminSession`: hashed admin session tokens and expirations.
- `AdminLoginAttempt`: login rate-limit tracking.
- `BannedIp`: blocked IP addresses and optional reasons.
- `PostReport`: deduplicated report records.
- `SpamSettings`: singleton moderation and spam-prevention settings.
- `AppSettings`: singleton app-level settings, currently Google Analytics.

Use migrations for schema changes:

```bash
npm run prisma:migrate
```

Generate the Prisma client after schema changes:

```bash
npm run prisma:generate
```

Production deploys should use:

```bash
npm run prisma:deploy
```

## Backup and Restore

Admins can download and restore the SQLite database from `/admin/settings`. This only works with direct SQLite file URLs such as `file:./dev.db` or `file:/data/prompt.db`.

Restore replaces the configured SQLite file on disk. Keep separate external backups before restoring production data.

## Resetting Admin Access

There is no email-based password reset because Prompt does not require SMTP. Reset an admin password from the terminal:

```bash
npm run admin:reset-password -- admin@example.com
```

The command prompts for a new password, requires at least 10 characters, hashes the password with the same `scrypt` storage used by the app, and clears existing sessions for that admin.

## Coolify and Docker Deployment

Prompt includes a Dockerfile suitable for Coolify. Use a Dockerfile-based app and expose port `3000`.

Recommended Coolify environment:

```bash
DATABASE_URL="file:/data/prompt.db"
PROMPT_ADMIN_SETUP_SECRET="use-a-long-random-secret"
PROMPT_TRUST_PROXY_HEADERS="true"
NEXT_PUBLIC_TURNSTILE_SITE_KEY="your-turnstile-site-key"
TURNSTILE_SECRET_KEY="your-turnstile-secret-key"
NODE_ENV="production"
PORT="3000"
```

In Coolify, keep `NODE_ENV=production` runtime-only. Do not mark it as
available at build time: npm skips `devDependencies` when `NODE_ENV` is
`production`, and the Docker build needs build-time packages such as
`@tailwindcss/postcss` for `npm run build`.

Add persistent storage before relying on production data. In Coolify, use
**Storages -> Add -> Directory Mount**:

```text
Source Directory: /data/coolify/applications/ic4kssssksswc4ocokkcgccg/data
Destination Directory: /data
```

The exact source directory can use any stable host path, but it must be outside
the disposable container filesystem. The destination must be `/data` because
`DATABASE_URL` points to `file:/data/prompt.db`.

If the app was already deployed before storage was mounted, copy the live
container database into the mounted source directory before redeploying:

```bash
ssh telepathy
mkdir -p /data/coolify/applications/ic4kssssksswc4ocokkcgccg/data
docker cp ic4kssssksswc4ocokkcgccg-212528356688:/data/prompt.db \
  /data/coolify/applications/ic4kssssksswc4ocokkcgccg/data/prompt.db
```

The container name changes after deploys. If needed, find the current container:

```bash
docker ps | grep ic4kssssksswc4ocokkcgccg
```

After adding storage and redeploying, verify the container has a mount:

```bash
docker inspect <container-name> --format '{{json .Mounts}}'
```

If `.Mounts` is empty, SQLite is still inside the disposable container and data
will be lost on the next redeploy.

The Docker image uses Node 22, installs dependencies with `npm ci`, builds the Next.js app, exposes port `3000`, and starts with:

```bash
npm run start:coolify
```

On startup, `scripts/coolify-start.sh` ensures the SQLite directory exists, runs `npm run prisma:deploy`, and starts Next.js on `0.0.0.0:$PORT`.

After the first deploy, visit `/admin/setup` and use the setup secret to create the first admin account.

## Security Notes

- User content is rendered as text, not raw HTML.
- Admin cookies are HTTP-only and use `secure` in production.
- Passwords are never stored directly.
- Session tokens are stored hashed in the database.
- Setup requires `PROMPT_ADMIN_SETUP_SECRET` in production.
- IP-based protections depend on trusted proxy header configuration.
- Database restore is powerful and should be available only to trusted admins.
